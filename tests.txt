Phase 0: Smoke check (now, ~2 min)

  Confirm the daemon came up clean.

  # Watcher started + connected to Lithos SSE + watch list loaded
  journalctl --user -u lithos-loom -n 200 2>/dev/null \
    || tail -200 ~/.local/share/lithos-loom/loom.log
  # (substitute whichever log destination you're using)

  # Grep for:
  #   "github-watcher child started"
  #   "github-watcher: watching N repo(s)"        ← N should be 3 (lithos-loom, lithos-core, influx)
  #   "github-watcher: loaded N cursor(s) and M stuck issue(s)"
  #   "LithosEventStream: connected"

  If the watch list is empty, run lithos-loom doctor — the project-context docs probably lost their github-watch tag.

  Phase 1: Lithos→GH push (~30 sec each, event-driven)

  These don't wait for the poll cycle. Best done with a throwaway test issue.

  Setup — create a test issue you can poke:

  gh issue create -R agent-lore/lithos-loom \
    --title "soak: slice 7.2 test $(date +%H%M)" \
    --body "Used for round-trip testing; ignore."
  # Note the issue number (e.g. #66).

  Wait ~10 sec, then verify the import happened:

  # Should appear in your Obsidian _lithos/tasks.md with #project/lithos-loom tag
  # Daemon log should show:
  #   "github-issue-sync: created task <id> for agent-lore/lithos-loom/#66"

  Test 1a — Lithos→GH title push. Rename the Lithos task via the MCP. Within ~5 sec (LithosEventStream emits task.updated):

  github-issue-push: renaming agent-lore/lithos-loom/#66 to 'new title' (task <id>)

  Verify on GH the title actually changed.

  Test 1b — Lithos→GH close mirror. Tick the task in Obsidian ([ ] → [x]). Within ~5 sec:

  github-issue-push: closing agent-lore/lithos-loom/#66 as completed (task <id>)

  gh issue view -R agent-lore/lithos-loom 66 should show closed-as-completed.

  Test 1c — Lithos cancel → GH close as not_planned. Recreate (or use another test issue), then [ ]→[-] in Obsidian:

  github-issue-push: closing agent-lore/lithos-loom/#NN as not_planned (task <id>)

  Phase 2: Periodic reconcile (~15 min, automatic)

  This is the slowest visibility loop you can confirm quickly. Wait until you see (at T+15 min from daemon start):

  github-watcher: reconcile sweep replayed N open / M completed / K cancelled GH-linked task(s)

  The counts are "tasks I considered", not "PATCHes made" — they should be roughly stable across consecutive sweeps. Crucially, between sweeps there should be
  zero github-issue-push: closing / renaming log lines (because the handler short-circuits when GH already matches).

  After two sweeps confirm stable behavior, optionally bump reconcile_interval_minutes back to 60.

  Phase 3: GH→Lithos polling (waits one poll = ~30 min)

  Your poll_interval_seconds = 1800, so each of these waits up to 30 min. Pick whichever you care about most or kick them all off in parallel and check the next
   morning.

  Setup — open a fresh test issue:

  gh issue create -R agent-lore/lithos-loom \
    --title "soak: GH→Lithos $(date +%H%M)" \
    --body "round-trip test\n\nbody line 2"
  # Wait one poll cycle (or check that the bootstrap walk caught it).

  Test 3a — title drift. Rename on GH:

  gh issue edit -R agent-lore/lithos-loom <num> --title "renamed on GH"

  After the next poll, the Lithos task title updates. Daemon log:

  github-issue-sync: drift-sync task <id>  (kwargs include title='renamed on GH')

  Test 3b — body drift. Edit body via web UI or gh issue edit --body. Same as 3a but description=. Verify the <!-- lithos:<id> --> marker is NOT in
  task.description.

  Test 3c — label drift. Add a label on GH; wait one poll. Check the Lithos task tags via MCP — new label should appear, any operator-added Lithos tags should
  still be there. Remove the label on GH; next poll removes from Lithos tags.

  Test 3d — GH close mirror. gh issue close <num> --reason completed. Next poll completes the Lithos task. gh issue close --reason "not planned" → cancelled.

  Phase 4: Reopen finding (after 3d)

  gh issue reopen -R agent-lore/lithos-loom <num>

  Next poll:

  github-issue-sync: reopen detected on agent-lore/lithos-loom/#NN (task <id>)

  Lithos task gets a [ReopenRequested] finding. Check the task — should have exactly ONE finding with that prefix even after multiple subsequent polls (dedup
  via metadata.github_state_snapshot).

  Phase 5: Exclude filters (~10 min if you want to exercise)

  Pick a project — influx is a good candidate if you don't want noise from automated issues there. Tag the project-context doc with:

  github-exclude-author:dependabot[bot]

  via MCP (no CLI command yet — tag-edit on the project-context note directly). Next poll, watch the watcher log for:

  github-watcher: watch list refresh — added=[] removed=[] retagged=['influx']
  github-watcher: resetting cursor for agent-lore/influx after watch-list change so re-included issues surface
  github-issue-sync: skipping agent-lore/influx/#NN on import (excluded author 'dependabot[bot]')

  Already-imported dependabot tasks survive (only filters at import time).

  Phase 6: Coord-doc inspection (passive, anytime)

  Verify cursor + stuck persistence:

  # Lithos MCP: read the coord doc
  # Should show rows like:
  #   agent-lore/lithos-loom 2026-05-30T18:42:11+00:00
  #   agent-lore/lithos      2026-05-30T18:42:11+00:00
  # Plus any stuck rows if dispatch failures occurred:
  #   stuck:agent-lore/lithos-loom#42

  The doc version should climb slowly (one bump per poll where cursors advanced), NOT every poll cycle — that was the round-2 regression.

  Phase 7: Idempotency soak (overnight)

  Leave the daemon running. In the morning, grep:

  grep "task_update\|github-issue-push: closing\|github-issue-push: renaming" loom.log | wc -l

  On a quiet repo with no GH activity, this count should track ONLY the events you triggered yourself. The reconcile sweeps and polls should not generate
  spurious PATCHes.

  What I'd watch for as red flags

  - Coord doc version climbing every minute (regression of round-2 bug) — lithos.note.updated events for the coord doc every poll.
  - [Friction] log lines accumulating without obvious cause — especially dispatch for X/#Y failed from the cursor-freeze path, or exhausted N attempts on from
  the push retry.
  - Duplicate Lithos tasks for the same GH issue URL — would indicate the _find_task_by_url recovery path broke (this is what the round-7 revert was about).
  - Stuck set growing unboundedly in the coord doc — implies a persistent dispatch failure that never recovers.

  grep "stuck" loom.log | tail -20
