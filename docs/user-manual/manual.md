# lithos-loom User Manual

lithos-loom is the workflow-orchestration tool that connects **Lithos** (your shared task and knowledge base) to coding agents. You run it from the command line: it watches Lithos for tasks, dispatches them to agents, mirrors your tasks into an Obsidian vault, and keeps GitHub issues in sync. This manual is for **operators** running lithos-loom on a host — for the internal design see [`docs/SPECIFICATION.md`](../SPECIFICATION.md).

## Overview

lithos-loom gives you three surfaces over the same set of Lithos tasks:

- **Obsidian bridge.** Your open Lithos tasks are projected into a file in your Obsidian vault (`_lithos/tasks.md`). Ticking a task complete, changing its priority, or editing its due date in Obsidian flows back to Lithos. Optional Templater macros let you create tasks and projects without leaving the editor.
- **Route-runner daemon.** `lithos-loom run` listens for Lithos task events, matches each task against the routes in your config by tag, claims matching tasks, and runs the matching plugin (for example, a story-develop agent that writes code). Results flow back to Lithos: completed tasks are marked done, failures are logged as findings.
- **GitHub issue watcher.** For projects you opt in, lithos-loom polls their GitHub repos, turns new and updated issues into Lithos tasks, and keeps the two in sync both ways.

There is no graphical interface — lithos-loom is a CLI plus a long-running daemon. Everything below is run from a terminal.

## Getting started

You need a running **Lithos** server and Python managed with [`uv`](https://docs.astral.sh/uv/).

1. **Bring up a Lithos server** (however you normally run it) and note its URL, e.g. `http://localhost:8765`.

2. **Drop a config in place** by copying the annotated example and editing it for your machine:

   ```bash
   mkdir -p ~/.config/lithos-loom
   cp examples/lithos-loom.toml ~/.config/lithos-loom/config.toml
   $EDITOR ~/.config/lithos-loom/config.toml
   ```

3. **Point loom at your Lithos server** by setting `LITHOS_URL`:

   ```bash
   cp .env.example .env
   $EDITOR .env            # set LITHOS_URL=http://localhost:8765
   ```

4. **Validate** the configuration before running anything:

   ```bash
   uv run lithos-loom validate-config
   uv run lithos-loom doctor
   ```

5. **Run the daemon** (it stays in the foreground; wrap it in tmux, nohup, or a systemd --user unit to keep it running):

   ```bash
   uv run lithos-loom run
   ```

With the daemon running you can create tasks (via the CLI, the Obsidian macros, or Lithos directly) and watch loom pick them up.

## Configuration

lithos-loom reads a single TOML file. It is located in this order:

1. the path in `LITHOS_LOOM_CONFIG`, if set;
2. otherwise, if `LITHOS_LOOM_ENVIRONMENT` is set (e.g. `dev`), `config.<env>.toml` in the current directory, then in `~/.config/lithos-loom/`;
3. otherwise `config.toml` in those same two locations.

You can always override the path per command with `--config/-c`. The one environment variable you must set is **`LITHOS_URL`** (your Lithos server). A fully annotated example lives at [`examples/lithos-loom.toml`](../../examples/lithos-loom.toml); the sections you set are:

- **`[orchestrator]`** *(required)* — `agent_id` (this host's identity), `lithos_url`, `work_dir`, `max_concurrency`, `log_level`, `retain_failed_workdirs`.
- **`[projects.<name>]`** — one block per project: `repo` (local working-tree path, required), and optional `claude_config` / `codex_config` agent profiles.
- **`[[routes]]`** — the dispatch table: `name`, `command` (the plugin to run, with `{{task_json}}` / `{{work_dir}}` / `{{result_file}}` / `{{repo}}` substitutions), `match.tags` (a task must carry all of them), `human_blocking`, `max_runtime_seconds`, `completes_task`.
- **`[[subscriptions]]`** — event handlers: `name`, `event_types` (or `on`), `action`, optional `match` / `where` filters, `retry.*`, and `on_persistent_failure`.
- **`[obsidian_sync]`** *(vault host only)* — `vault_path` (required), `tasks_file`, `projects_dir`, `resolved_ttl_days`, `include_blocked`, `exclude_tags`.
- **`[github_watcher]`** — `enabled`, `poll_interval_seconds`, `reconcile_interval_minutes`, `pr_merge_poll_enabled`, and related cursor settings.
- **`[story_develop]`** — host-wide defaults for the story-develop plugin: `operator_github_login`, `default_review_profile`, and per-tool `[story_develop.default_models]`.

Use `lithos-loom validate-config` to type-check the file and `lithos-loom config --show` to print the merged result.

#### `lithos-loom task create`

Create a Lithos task and print its projected task line.

```text

#### `lithos-loom develop attach`

Follow a live run until it finishes; print hand-offs and the outcome.

```text
 Usage: lithos-loom develop attach [OPTIONS] KEY

 Follow a live run until it reaches a **terminal state**, printing handoffs as they land
 plus the current round + active agent, then a one-line outcome summary. Following keys on
 terminal state, not agent liveness, so it spans both the startup window before the first
 container and the commit / test-gate / teardown after the last agent turn, grace-polling
 through the window where the plugin has stopped its containers but not yet written the
 outcome. An **approved** verdict is not yet the end in daemon mode — PR delivery (push +
 Copilot round + ``result.json``) runs after the dialogue approves, shown as a distinct
 "delivering PR…" phase — so attach follows through it instead of exiting early. If the
 work dir is reaped on success before a poll observes the result, the outcome is recovered
 from the plugin's completion store. Read-only; ``Ctrl-C`` exits cleanly. When docker is
 unavailable it still follows the handoff files (active agent shows as ``—``).

 first until the run appears (so it can be used immediately after dispatch),

│ --config  -c      PATH  Explicit TOML config path.                                       │
│ --once                  Print one snapshot and exit (no follow).                         │
│ --wait                  Block silently until the run reaches a terminal state, then      │
│                         print only the outcome (exit non-zero unless approved).          │

## Common workflows

**Set up a new host.** Copy the example config, set `LITHOS_URL`, then verify before running:

```bash
cp examples/lithos-loom.toml ~/.config/lithos-loom/config.toml && $EDITOR ~/.config/lithos-loom/config.toml
uv run lithos-loom validate-config && uv run lithos-loom doctor
```

**Create or import a project.** Make an empty project, or import an existing Markdown plan (its `- [ ]` lines become tasks):

```bash
uv run lithos-loom project create --title "My Project"
uv run lithos-loom project import plan.md --slug my-project
uv run lithos-loom project list            # see all projects + local status
```

**Create a task.** Tag it so a route can match it:

```bash
uv run lithos-loom task create --project my-project --title "Add health endpoint" --tags backend,ready
```

**Run and monitor the daemon.** Run loom in tmux, then watch story-develop runs as they happen:

```bash
uv run lithos-loom run                      # in its own tmux window
uv run lithos-loom develop list             # what's in flight
uv run lithos-loom develop attach <run-id> --wait
uv run lithos-loom develop dump  <run-id>   # full conversation log
```

**Troubleshoot.** If tasks aren't being picked up or the vault looks wrong:

```bash
uv run lithos-loom validate-config --dry-run    # simulate which routes/subscriptions would fire
uv run lithos-loom doctor                        # vault + project-config checks
uv run lithos-loom develop prune --dry-run       # see stale run dirs before cleaning
```
