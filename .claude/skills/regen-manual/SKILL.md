# Regenerate the user manual

You are regenerating the end-user manual for **lithos-loom** from scratch. Output: `docs/user-manual/manual.md`.

lithos-loom is a command-line tool (a Typer CLI plus a background daemon) with **no graphical UI**, so the manual's captured artifact is **real terminal output** — the verbatim `--help` text of every command — embedded in fenced code blocks. There are no screenshots.

This is a full rewrite. Do **not** read the previous `manual.md` and do **not** diff against it — the bundled template is the single source of truth for structure. Just produce the new manual.

## Pre-flight (abort if it fails)

Confirm the CLI is runnable (it does not need any services for `--help`):

```bash
uv run lithos-loom --help >/dev/null 2>&1 && echo OK || echo FAIL
```

If that prints anything other than `OK`, abort with this message and STOP:

> Cannot regenerate the manual: `lithos-loom` is not runnable. Run `uv sync` first, then try again.

## Step 1 — Load the template

Read the bundled template `.claude/skills/regen-manual/manual-template.md`. It defines the exact shape of the output: the title block, the intro, the "do not edit by hand" note, the Contents list, the prose sections (Overview, Getting started, Configuration, Common workflows), and the **command-reference pattern** (one `### lithos-loom <command>` subsection per command, each with a one-sentence "what it's for" line followed by a fenced ` ```text ` block holding that command's captured `--help`). Every section you write must follow that pattern.

## Step 2 — Discover the live command inventory

Enumerate the command tree from the running CLI (do not hardcode it):

```bash
uv run lithos-loom --help          # top-level commands + groups
uv run lithos-loom task --help     # repeat for each group: task, project, obsidian-sync, develop, eval
```

Cross-check against the source so you miss nothing: read `src/lithos_loom/main.py` (the root `typer.Typer()` and its `add_typer(...)` calls) and the files under `src/lithos_loom/cli/`. The groups are the `add_typer` names; the commands within each group are its `@app.command()` functions.

## Step 3 — Capture each command's help

For every command and subcommand in the tree, capture its real help text at a fixed width so the output is stable:

```bash
COLUMNS=92 uv run lithos-loom <command path> --help 2>/dev/null
```

(Use `2>/dev/null` to drop uv's build chatter; capture stdout only. Strip trailing whitespace from each line.) The captured block for each command goes verbatim into a ` ```text ` fence under that command's `###` heading — this is the manual's equivalent of a screenshot.

You MAY additionally capture the output of **read-only** commands that are safe offline (e.g. `lithos-loom project list --source toml --format text`, `lithos-loom config --show`) to show example output, but never run mutating commands (`task create`, `project create/import/regenerate-done`, `run`) or anything that contacts a live Lithos/GitHub during regeneration.

## Step 4 — Write `manual.md`

Write `docs/user-manual/manual.md` from scratch, following the bundled template exactly.

Top of `manual.md` must contain:

1. `# lithos-loom User Manual`
2. A one-paragraph intro: what the tool is for, plus the "do not edit by hand — regenerated by `/regen-manual`" line.
3. A Contents list linking to each `##` section.

Then, in order: an **Overview** (the three surfaces — Obsidian bridge, route-runner daemon, GitHub issue watcher), **Getting started** (install → configure → validate → run, grounded in `README.md`), **Configuration** (the `config.toml` sections from `src/lithos_loom/config.py` and `examples/lithos-loom.toml`, plus `LITHOS_URL` and config precedence), the **Command reference** (one `## <group>` per command group with a `###` per command and its captured `--help`), and **Common workflows** (set up a host, create/import a project, create a task, run + monitor the daemon with `develop list/attach/dump`, troubleshoot with `doctor`/`validate-config`).

Tone: instructional second-person ("Run `lithos-loom doctor` to check your vault is writable..."). Audience: **operators running lithos-loom**, not contributors. Do NOT document internal modules, the event-bus implementation, or other developer concerns — point such readers to `docs/SPECIFICATION.md` instead.

## Step 5 — Verify completeness

Every command discovered in Step 2 must have a `###` section in the reference, and there must be no section for a command that no longer exists. Re-list the tree and diff it against your headings before finishing.
